These use cases were chosen because they: 
 Cover the primary operational flows of the system (registration, appointment 
management, and billing). 
 Involve rich user–system interaction, which is essential for menu design, screen layout, 
and interaction flow design. 
 Map directly to the key domain classes identified in Deliverable 3 (Patient, Doctor, 
Appointment, MedicalRecord, Room, Bill, Prescription). 
 They are already supported by activity diagrams, system sequence diagrams, and state 
machines, ensuring traceability between analysis and design. 
 
 UC-01 – Book appointment 
 Primary actor: Patient 
 Supporting actors: HealthcareSystem, optionally Receptionist 
 Brief description: 
The patient (or the receptionist on behalf of the patient) searches for available doctors 
and time slots, selects a preferred slot, and submits a booking request. The system 
validates the slot, reserves the corresponding room/slot, and creates an 
Appointment. Depending on clinic policy, the system may also initiate pre-payment as 
part of the booking process. 
 Rationale for inclusion as a core use case: 
Booking appointments is central to the healthcare system’s purpose. This use 
case links several core domain classes (Patient, Doctor, Appointment, Room) 
and contains non-trivial decision logic (slot availability, alternative proposals, optional 
pre-payment). It is therefore a key driver for designing intuitive booking screens, filters, 
and confirmation views in the user interface

UC‑02 – Register patient information 
 Primary actor: Receptionist 
 Supporting actors: HealthcareSystem 
 Brief description: 
 The receptionist opens the patient registration function, enters or updates patient 
demographic and clinical details, and submits the information. The system validates the 
data and, on success, creates or updates the corresponding Patient and MedicalRecord 
entries, returning a confirmation message. In case of invalid data, the system returns 
detailed validation errors for correction. 
 Rationale for inclusion as a core use case: 
 Patient registration is a prerequisite for almost all other operations in the system, 
including appointment booking, treatment documentation, and billing. This use case 
exercises form‑based data entry, validation feedback, and usability aspects of the 
interface. It is directly connected to the core domain classes Patient and MedicalRecord 
defined in Deliverable 3. 
 
UC-03 – Manage doctor schedule 
 Primary actor: Doctor 
 Supporting actors: HealthcareSystem 
 Brief description: 
The doctor accesses their schedule, views existing time slots, and performs updates such 
as adding new availability, blocking time ranges, or modifying existing slots. The 
system checks for conflicts (e.g., overlapping slots, invalid time ranges) and either 
applies the changes or returns detailed conflict information for correction. 
 Rationale for inclusion as a core use case: 
The availability managed in this use case directly determines which appointment 
slots can be booked or cancelled. It requires clear visualization of time-based data 
(e.g., calendars, tables), as well as intuitive editing and conflict-resolution feedback. The 
use case is tightly connected to the Appointment lifecycle and the scheduling/slot 
concepts in the domain model. 
UC‑06 – Cancel appointment 
 Primary actor: Patient or Receptionist 
 Supporting actors: HealthcareSystem, NotificationService 
 Brief description:

 The actor selects an upcoming appointment and requests its cancellation. The system 
evaluates the applicable cancellation policy (for example, whether the request is more 
than 24 hours before the appointment). If allowed, the system updates the Appointment 
status (e.g., Cancelled or Late‑cancelled), releases the associated room/slot, and sends 
notifications to the patient and doctor. If the policy is violated, the system may show 
warnings or penalties before final confirmation. 
 Rationale for inclusion as a core use case: 
 Appointment cancellation is a frequent real‑world operation with important policy rules 
and side effects. The user interface must support clear appointment selection, explanation 
of policy constraints, and confirmation dialogs. This use case is closely related to the 
Appointment lifecycle, resource reuse (Room/slot), and notification behavior. 
 
UC-07 – Make payment 
 Primary actor: Patient 
 Supporting actors: HealthcareSystem, external Payment Gateway 
 Brief description: 
The patient views a list of unpaid bills, selects a bill to settle, chooses a payment method, 
and confirms the payment. The HealthcareSystem sends a payment request to the external 
Payment Gateway and receives an approval or failure response. On approval, the system 
updates the Bill status to Paid, records transaction details, and shows a 
success confirmation. On failure, it displays an error message and may offer options to 
retry or cancel the attempt. 
 Rationale for inclusion as a core use case: 
This use case is central to the financial processes of the system. It directly interacts with 
the Bill class and the Bill lifecycle state machine (PendingPayment, Paid, Failed, 
Overdue). The user interface must clearly display bill details, payment options, and 
error/retry behavior. As a result, it provides a strong basis for designing payment screens, 
confirmation dialogs, and integration feedback forms. 
 
 
Assumptions 
The following assumptions underpin the design decisions in this deliverable:

 User Roles: 
The system supports three primary roles—Patient, Receptionist, and Doctor—each with 
role-specific dashboards and permissions. 
 External Service Availability: 
External integrations (Payment Gateway, Notification Service, Reporting Service, 
External Patient System) are assumed to be available via secure APIs. If unavailable, the 
system logs failures and continues the main workflow where safe. 
 Cancellation Policy: 
Appointment cancellation is governed by a time-based threshold (e.g., 24 hours), which 
may trigger warnings or penalties. The UI includes appropriate confirmation dialogs. 
 Reporting Output: 
Reports are generated electronically (PDF or Excel format). External distribution requires 
user consent and audit logging. 
 Java GUI Target: 
The user interface is intended to be implemented using Java GUI frameworks (Swing or 
JavaFX), aligned with the A4 architecture. 
 Traceability: 
All UI and system interface designs remain traceable to Deliverable 3 artifacts, including 
use cases, system sequence diagrams (SSDs), activity diagrams, sequence diagrams, and 
state machines.

Overall Menu and Navigation Concept 
The user interface of the Healthcare Management System follows a role-based main window 
with a shared top menu bar and contextual navigation panels. The goal is to 
provide a consistent structure for all actors (Patient, Receptionist, Doctor) while keeping 
frequently used tasks easily accessible for each role. 
Main Window Structure 
After successful login via LoginFrame, the system opens a role-specific dashboard frame: 
 PatientDashboardFrame for patients 
 ReceptionistDashboardFrame for receptionists 
 DoctorDashboardFrame for doctors 
Each dashboard is implemented as a main window that contains: 
 A top menu bar with global menus (e.g., Home, Appointments, Patients, Billing, Help). 
 A left side navigation panel (or tab bar) showing shortcuts to the most relevant 
functions for that role. 
 A central content area where panels such 
as BookAppointmentPanel, RegisterPatientFormFrame, DoctorScheduleManagementFra
me, or PaymentOverviewFrame are displayed.

UI/UX Design Guidelines for Java GUI and Figma 
This section defines the common visual and interaction guidelines for both the Java-based 
graphical user interface and the Figma prototypes of the Healthcare Management System. The 
goal is to ensure that all screens and storyboards produced by different team members 
remain visually consistent, usable, and aligned with the clinical context of the system. 
Visual Theme and Color Palette 
Overall Theme 
 The application uses a light theme with high contrast and a clean, minimalist layout 
suitable for healthcare environments. 
 Screens should avoid visual clutter and prioritize readability and clarity over decorative 
elements. 


===== PAGE 31 =====

Introduction to software engineering project 
Deliverable four

Color Palette 
 Primary color: 
 Medical blue, e.g., #1E88E5. 
 Usage: primary buttons (e.g., “Save”, “Confirm Payment”), selected menu items, 
key highlights and links. 
 Secondary color: 
 Neutral dark gray, e.g., #424242. 
 Usage: labels, secondary buttons, icons, borders, and non-critical elements. 
 Accent colors for feedback: 
 Success: Green, e.g., #43A047. 
 Usage: success messages (“Appointment booked”, “Payment completed”), 
success icons or checkmarks. 
 Error / Warning: Red, e.g., #E53935. 
 Usage: validation errors, important warnings (late cancellation penalties, 
payment failures), error icons. 
 Backgrounds: 
 Main window background: light gray or off-white, e.g., #F5F5F5. 
 Content panels, cards, and dialogs: white, #FFFFFF. 
 Tables and forms may use subtle zebra striping or light borders to separate rows 
without heavy lines. 
 Buttons, Forms, and V alidation 
Button Styles 
 Primary buttons: 
 Background: primary blue (#1E88E5). 
 Foreground: white. 
 Used for the main action on the screen: “Save”, “Book Appointment”, “Confirm 
Payment”. 
 Only one primary button per main action area when possible. 
 Secondary buttons:

 Foreground: dark gray. 
 Used for actions like “Cancel”, “Back”, “Reset Form”. 
 Destructive actions (e.g., “Cancel Appointment”): 
 Background: red (#E53935) or text in red with clear warning icon. 
 Require confirmation dialogs for irreversible actions. 
Form Fields and Validation 
 All input fields should have: 
 Clear, non-ambiguous labels. 
 Mandatory fields marked with an asterisk () and explained in a small legend 
(“ required”). 
 Validation behavior: 
 When invalid input is submitted: 
 The specific field is highlighted (e.g., red border). 
 A short message is shown near the field (e.g., “Please enter a valid date”). 
 A general error summary may appear at the top of the form if multiple 
fields are invalid. 
 Examples: 
 On RegisterPatientFormFrame, missing name or invalid date of birth should 
highlight those fields and show concise hints. 
 On PaymentProcessingDialog, invalid card data or missing method should result 
in clear error messages. 
 Feedback, Status, and Error Messages 
 Success messages: 
 Use green text or icons with concise content. 
 Examples: “Appointment booked successfully.”, “Payment completed.” 
 Info/warning messages: 
 Use neutral or orange/yellow warning styles for non-fatal issues.
